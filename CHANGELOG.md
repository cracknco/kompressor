### Changed (BREAKING)

* **api (all platforms):** **retire the path-based `compress(inputPath: String, outputPath: String, config, onProgress: suspend (Float) -> Unit)` overload** on `ImageCompressor` / `AudioCompressor` / `VideoCompressor`. * **io (android):** **Fraction-accurate `MATERIALIZING_INPUT` progress for `ParcelFileDescriptor` inputs.** `AndroidMediaDispatch.materializePfdHandle` now routes the PFD тЖТ temp-file copy through the shared CRA-91 `TempFileMaterializer` primitive instead of a raw `FileInputStream.copyTo(FileOutputStream)`, and seeds the `sizeHint` via `estimateSourceSize(AndroidPfdMediaSource(pfd))` тАФ which activates the `AndroidPfdMediaSource` branch of the CRA-96 `SourceSizeProbe` that reads `ParcelFileDescriptor.getStatSize()` (floor-clamped to `null` for `-1` sentinel values on pipe / socket-backed FDs). Paired with the invariant helper are `ProgressionE2ETest` suites on both platforms (`androidDeviceTest` + `iosTest`) covering audio ├Ч video ├Ч `{FilePath, Stream, Bytes}` (six tests each) that additionally assert `COMPRESSING(0f)` is emitted at compression start and `FINALIZING_OUTPUT(1f)` is the terminal event тАФ the canonical "UI: render 0%" / "UI: hide the progress bar" signals. `ProgressionInvariantsTest` in `commonTest` pins the invariant helper itself against 9 hand-crafted sequences (canonical full sequence, skipped-materialization sequence, repeated-fraction-within-phase, regression rejections). **Infrastructure:** new `internal expect fun estimateSourceSize(input: MediaSource.Local): Long?` (actuals in `androidMain` + `iosMain`) ships the platform size-probe surface тАФ `FilePath` via `File.length()` / `NSFileManager[NSFileSize]`, `AndroidUriMediaSource` via `ContentResolver.query(OpenableColumns.SIZE)` + `openFileDescriptor.statSize` fallback, `AndroidPfdMediaSource` via `ParcelFileDescriptor.statSize` (`-1` тЖТ null), `IosUrlMediaSource` via `NSFileSize`, `IosPHAssetMediaSource` via the **primary-resource-only** `PHAssetResource.valueForKey("fileSize")` selected by media type (Photo / Video / Audio) тАФ avoids over-reporting on Live Photos + edits + adjustment sidecars. Every platform branch is individually `runCatching`-guarded so a `SecurityException` on a revoked URI, a deleted file, or a missing KVC key surfaces as a silent `null` rather than aborting the compression. The probe is currently wired only into `MediaSourceResolver.materializeStream` as `input.sizeHint ?: estimateSourceSize(input)` тАФ which is a **no-op for Stream inputs** because the Stream branch is a passthrough of `sizeHint`. No public API surface added (every new type is `internal`); ABI baseline unchanged. ### Added

* **image (all platforms):** new `ImageCompressor.thumbnail(input, output, maxDimension, format, quality)` entry point that produces memory-efficient thumbnails via sampled decode on both platforms. Android routes through a 2-pass `BitmapFactory` pipeline (`inJustDecodeBounds=true` to read source dims, power-of-two `inSampleSize` selected by the existing `calculateInSampleSize` heuristic for the subsampled decode, then exact-resize to the final target via `Bitmap.createScaledBitmap`) тАФ peak heap stays bounded (~1тАУ2 MB for a 4000├Ч3000 source subsampled to 500├Ч375 + ~120 KB for the 200├Ч150 exact-resize output) regardless of source megapixels. iOS routes through `CGImageSourceCreateThumbnailAtIndex` with `kCGImageSourceThumbnailMaxPixelSize = maxDimension`, `kCGImageSourceCreateThumbnailFromImageAlways = true` (force decoding instead of returning an embedded thumbnail that may have unknown quality), and `kCGImageSourceCreateThumbnailWithTransform = true` so EXIF orientation is applied inside the sampled decode тАФ a 200├Ч100 landscape JPEG with `kCGImagePropertyOrientation = 6` (Right, 90┬░ CW) thumbnails to 100├Ч200 portrait in one pass. Contract is orientation-symmetric and never upscales: when `maxDimension тЙе max(source)` the output keeps source pixel dimensions (re-encoded to format/quality); when `maxDimension < max(source)` the longer edge is clamped to `maxDimension` and the shorter edge scales proportionally, preserving aspect ratio to 1 px on a 4:3 source. Fail-fast validation: `maxDimension тЙд 0` returns `Result.failure(IllegalArgumentException)` at the entry point without touching the destination file on either platform. Dispatch is unified through `AndroidMediaDispatch.toAndroidInputPath` / `IosMediaDispatch.toIosInputPath` so the same `MediaSource` / `MediaDestination` surface introduced in CRA-89 (Uri / PFD / Stream / Bytes on Android; NSURL / PHAsset / Stream / NSData on iOS) works as input/output тАФ not just `FilePath`. Observability: both implementations wrap the pipeline in the CRA-47 `logger.instrumentCompress(LogTags.Image, "thumbnail", ...)` lifecycle so the INFO start + INFO success + ERROR on failure trio emits through `KompressorLogger` identical to the `compress()` call path. Coverage: 7 new contract tests in `commonTest/image/ImageThumbnailContractTest` (dimension arithmetic invariants on `calculateTargetDimensions` + `calculateInSampleSize` тАФ the heuristic uses a `halfDim / inSampleSize тЙе target` gate that stops one step earlier than a naive `decodedDim тЙе target` would, returning 8 for 4000├Ч3000 тЖТ 200├Ч200 because `1500 / 16 = 93 < 200`; Pass 2 then finishes the last mile from 500├Ч375 to 200├Ч150), 5 property tests in `commonTest/property/ImageThumbnailPropertyTest` (Kotest `checkAll` over randomized source dims 100..8000 ├Ч maxDim 50..1000 with fixed seed тАФ longer edge never exceeds maxDim, output dims always positive, no upscale, `calculateInSampleSize` is always a power-of-2 тЙе 1), 6 device tests in `androidDeviceTest/ImageThumbnailDeviceTest` (4000├Ч3000 PNG тЖТ 200├Ч150 JPEG with aspect preserved; peak heap delta тЙд 10 MB during sampled decode тАФ a regression flipping back to full-resolution decode would hit ~48 MB and fail the envelope; EXIF ORIENTATION_ROTATE_90 on a 200├Ч100 landscape source produces 100├Ч200 portrait output proving the transform fires inside the decode path; `maxDimension=0` / `-1` return `Result.failure(IllegalArgumentException)` without creating the output file; source already smaller than `maxDimension` keeps source pixel dimensions), 5 simulator tests in `iosTest/ImageThumbnailIosTest` mirroring the Android matrix (fixture helper `createExifTaggedJpeg` builds a `CGBitmapContext`-backed JPEG with an embedded `kCGImagePropertyOrientation` dictionary entry; `kCGImageSourceCreateThumbnailWithTransform=true` swaps oriented dims to 100├Ч200 for orientation=6, a regression dropping the transform flag would produce 200├Ч100). **Android/iOS parity:** complete тАФ identical public signature, identical behaviour on every DoD invariant, sibling device/simulator suites per platform. ABI baseline updated via `apiDump` тАФ three new public methods (`thumbnail-hUnOzRk` on the `ImageCompressor` interface + `$default` synthetics) [CRA-108]

* **io (all platforms):** wire `MediaSource.Local.Stream` / `MediaSource.Local.Bytes` / `MediaDestination.Local.Stream` тАФ plus the CRA-94 iOS-only `IosDataMediaSource` (`NSData`) тАФ through every one of the six platform compressors (`AndroidImageCompressor` / `AndroidAudioCompressor` / `AndroidVideoCompressor` / `IosImageCompressor` / `IosAudioCompressor` / `IosVideoCompressor`), completing the M10 I/O epic started in CRA-89. Image compressors short-circuit `Bytes` / `NSData` directly to platform-native in-memory decoders тАФ Android `BitmapFactory.decodeByteArray`, iOS `UIImage(data:)` / `CGImageSourceCreateWithData` тАФ so the image path never materialises to disk and resident heap stays at `O(bytes.size)` instead of `O(2 ├Ч bytes.size)`; image `Stream` inputs also read-to-byte-array in-place rather than round-tripping a temp file. Audio + video compressors route `Stream` / `Bytes` / `NSData` through a new `commonMain` `resolveStreamOrBytesToTempFile(input, tempDir, mediaType, logger, onProgress)` helper that chunked-copies to a UUID-named temp file (`<tempDir>/kmp_io_<uuid>.bin`) via the CRA-91 `TempFileMaterializer` primitive тАФ O(64 KB) heap regardless of clip size, cooperative cancellation on every chunk boundary, temp-file cleanup on any throwable. Materialisation progress surfaces through the CRA-90 `CompressionProgress.Phase.MATERIALIZING_INPUT` phase; the underlying transcode still emits `COMPRESSING` ticks; a terminal `Phase.FINALIZING_OUTPUT(1f)` fires once the compress call returns success. `MediaDestination.Local.Stream` writes first to a temp file (same `<tempDir>/kmp_io_<uuid>.bin` convention) and then chunk-copies to the caller-provided okio `Sink` via a new `commonMain` `OutputSink.writeTempFileToSink(tempFile, sink, onProgress)` helper, emitting a second pass of `FINALIZING_OUTPUT` fractions during the copy so UI progress bars don't jump from 99% to 100% in one tick for multi-GB outputs. `closeOnFinish` is honoured symmetrically on both sides: `MediaSource.Local.Stream.closeOnFinish=true` closes the underlying `okio.Source` in the resolver's cleanup closure; `MediaDestination.Local.Stream.closeOnFinish=true` closes the caller's `okio.Sink` after the copy completes тАФ both guarded inside `runCatching` so a misbehaving stream can't mask the real compress result. OOM guard-rail: `Bytes` / `NSData` inputs whose `size > 10 MB` (new `BYTES_WARN_THRESHOLD` constant in `MediaSourceResolver.kt`) emit a WARN through the CRA-47 `SafeLogger` pipeline under `LogTags.IO` тАФ images are exempt (they short-circuit and never materialise); audio/video WARN carries the exact byte count + recommends switching to `FilePath` / `Uri` / `NSURL` / `Stream`. **iOS container-UTI fix:** new `renameToAvRecognizableExtension(resolved, mediaType)` step in `IosMediaDispatch.kt` тАФ after materialisation, peeks 16 bytes from the temp file (`try { src.read(buf, MAGIC_BYTE_PEEK) } finally { src.close() }` because `okio.Source` is not `AutoCloseable` on Kotlin/Native, so `.use {}` doesn't compile), matches magic bytes against `RIFF`+`WAVE` (`.wav`) / `ftyp` at offset 4 (`.m4a` / `.mp4`) / `ID3` (`.mp3`), and `atomicMove`s the file to the correct extension before handing it to `AVURLAsset`. Without this step `AVURLAsset` derives the container UTI from `pathExtension` тАФ `.bin` maps to the generic `public.data` UTI, `AVAsset.tracks` returns empty, and the compressor throws `UnsupportedSourceFormat` on valid audio/video bytes. * **internal (all platforms):** introduce `TempFileMaterializer` тАФ an internal `suspend fun Source.materializeToTempFile(tempDir, sizeHint, onProgress)` primitive under `co.crackn.kompressor.io` that chunk-copies an okio `Source` to a temp file (`<tempDir>/kmp_io_<uuid>.bin`) with a 64 KB buffer so resident heap stays O(64 KB) regardless of stream size (validated by a 50 MB Android device test that asserts a < 10 MB native-heap delta via `Debug.getNativeHeapAllocatedSize()`). `currentCoroutineContext().ensureActive()` is queried on every chunk boundary so callers get cooperative cancellation; both the `CancellationException` path and any other `Throwable` from the copy loop trigger `fileSystem.delete(tempFile, mustExist = false)` before rethrowing the original exception, so callers never observe a partial temp file on the failure path. The receiver `Source` is **not** closed тАФ the caller owns the lifecycle (honours the `MediaSource.Local.Stream.closeOnFinish = false` contract introduced in CRA-90). Progress is emitted per chunk as a fraction `bytesWritten / sizeHint` clamped to `[0f, 1f]` (survives a caller that underestimates the real source size); when `sizeHint` is `null` or `0L`, `onProgress` still fires at every chunk but with a constant `0f` so callers can at least count invocations as a "something is happening" heartbeat.
