## [Unreleased]

### Added

* **logging (all platforms):** pluggable `KompressorLogger` sink — the library never writes to `println` / `android.util.Log` / `NSLog` directly anymore. Every diagnostic emission goes through a public `fun interface KompressorLogger.log(level, tag, message, throwable)` with levels `VERBOSE` / `DEBUG` / `INFO` / `WARN` / `ERROR`. `createKompressor()` gains an optional `logger: KompressorLogger = PlatformLogger()` parameter (source-compatible — existing call sites keep working); `PlatformLogger()` is the `expect`/`actual` default that dispatches to `android.util.Log.{v,d,i,w,e}` on Android and `NSLog` on iOS (NSLog is chosen because `os_log` is a C macro that Kotlin/Native can't call directly — a Swift bridge recipe for `os_log` / SwiftLog / CocoaLumberjack is documented in `docs/logging.md`). Opt-in silence via the new `NoOpLogger` object. `KompressorLogger` also exposes an optional `isEnabled(level): Boolean = true` hook: simple loggers keep dispatching everything (baseline unchanged), but delegates that statically silence a level (`NoOpLogger`, production threshold loggers) override it so the library skips the lazy-message lambda entirely — truly allocation-free even at `VERBOSE` / `DEBUG` on hot paths. Library emissions are tagged with five constants from `LogTags` (`Kompressor.Init` / `.Probe` / `.Image` / `.Audio` / `.Video`, all ≤ 23 chars for Android API < 26), so a single `Kompressor.*` filter captures everything in Logcat / Console.app / SwiftLog. Lifecycle is uniform across image / audio / video on both platforms via `SafeLogger.instrumentCompress` (INFO start + INFO success + ERROR on failure; `CancellationException` is forwarded without an ERROR log — cancellation is a cooperative signal, not a library failure). `SafeLogger` wraps every emission in a catch-all so a misbehaving consumer logger (disk-full, serialisation bug) can't crash a transcode mid-pipeline. Previously ad-hoc `Log.w` / raw-log call sites in the video and audio pipelines were replaced with structured `logger.debug(LogTags.*)` / `logger.warn(...)` calls; every pipeline-decision branch (AAC passthrough vs transcode, `AVAssetExportSession` vs `AVAssetWriter`) now emits a DEBUG line, and probe() failures emit a WARN with the cause attached. Two CI gates enforce the "no raw logs in the library" invariant: Detekt `ForbiddenImport` blocks `android.util.Log` imports everywhere in `kompressor/src` except the Android `PlatformLogger` actual, and `scripts/check-no-raw-logging.sh` (wired as the new `no-raw-logging` job in `pr.yml`, ~10 s, no Gradle) greps for `println(` / `print(` / `android.util.Log.*` / `NSLog(` across the module with the two platform actuals allowlisted. Contract + integration recipes for Timber (Android), SwiftLog / CocoaLumberjack / `os_log` (iOS) live in the new `docs/logging.md`; the level taxonomy, tag vocabulary, no-throw rationale, and alternatives considered (depend-on-Kermit, `sealed class` sink, library-side level filter) are documented in the new `docs/adr/003-logger-contract.md`. Common + per-platform coverage in new `LoggerContractTest` (commonTest — partitioned-recorder thread-safety + no-throw + custom-injection) and `LoggerInjectionTest` (androidHostTest + iosTest — `createKompressor(logger=recorder)` end-to-end proof that the injected logger receives the tagged lifecycle records). Public API additions are source-compatible and tracked in the regenerated `kompressor/api/kompressor.api` baseline [CRA-47]

* **video+audio (iOS):** extend the `AVAssetExportSession` fast path in `IosVideoCompressor` / `IosAudioCompressor` to cover HDR10 HEVC video configs and non-default `audioTrackIndex` inputs. Video: the eligibility predicate now admits `VideoCompressionConfig(codec = HEVC, dynamicRange = HDR10)` at defaults and picks `AVAssetExportPresetHEVCHighestQuality` (preserving BT.2020 primaries + SMPTE ST 2084 PQ transfer + BT.2020 YCbCr matrix when the source is HDR10); every other knob — custom `videoBitrate`, non-default `maxResolution`, SDR HEVC — still falls back to the custom `AVAssetWriter` transcode pipeline. Audio: `canUseAudioExportSession` drops the `audioTrackIndex == 0` guard and now keeps any non-default track index on the preset path; `IosExportSessionPipeline` builds an `AVMutableAudioMix` with per-track volume muting (`selected → 1.0`, others `→ 0.0`) so the preset emits just the requested track instead of summing all source audio tracks. The HEVC Main10 capability is still gated by the pre-existing `requireHdr10HevcCapability` probe; iOS 15 is the library's min target so the DoD's "fallback if iOS < 15" requirement is auto-met. Eligibility + AVMutableAudioMix behaviour are covered by the new `iosTest/IosExportSessionFastPathTest` (video predicates on MediumQuality / HEVCHighestQuality / fallback cases; audio preset-path track selection verified end-to-end via Goertzel dominant-frequency analysis on a two-track stereo MP4 fixture); the HDR10 HEVC round-trip on physical A10+ hardware lives in `iosTest/Hdr10ExportRoundTripTest`. `MultiTrackAudioFixture.createMultiTrackAudioMp4` gains a `channelsPerTrack` parameter (defaults to 1, added 2 for stereo) with a constant-tone-across-channels stereo WAV writer, since `WavGenerator.generateWavBytes` applies a per-channel frequency multiplier that is wrong for tests that rely on a single dominant peak after mono downmix [CRA-19]

* **video (Android):** active `MediaCodec` runtime probe for HEVC Main10 / HDR10 encode support, replacing the self-declared `MediaCodecList` check as the final gate. `Hdr10HevcProbe` (`androidMain/video/`) picks the first HEVC encoder advertising `HEVCProfileMain10` + `FEATURE_HdrEditing`, allocates it via `MediaCodec.createByCodecName`, configures a 1×1 P010 + BT.2020 + ST 2084 (PQ) format, queues one probe frame, and drains until the first output buffer or output-format change — so OEM firmwares that *advertise* Main10 but crash `configure()`/first-frame are caught up-front instead of mid-export. The capability-matrix check remains as a ~1 ms cheap-path short-circuit; the active probe only runs when the capability check passes. Verdict cached in `SharedPreferences("co.crackn.kompressor.hdr10")` keyed by `"${Build.MODEL}|${codecName}"` so only the first HDR10 call per install pays the ~100–300 ms cost. Exposes new public boolean `probeHdr10HevcSupport()` (via `Hdr10HevcProbe`) and new typed error `VideoCompressionError.Hdr10NotSupported(device, codec, reason)` that callers can `when`-branch to offer an SDR fallback instead of a generic `UnsupportedSourceFormat`. Device-side validation in new `Hdr10ProbeActiveTest` (`androidDeviceTest`) asserts determinism + capability-matrix short-circuit + SharedPreferences cache-hit latency. iOS already had its equivalent preflight via `AVAssetWriter.canApplyOutputSettings`; this closes the parity gap [CRA-20]

* **docs:** `docs/format-support.md` format-support matrix — three tables (image / audio / video) answering the "does Kompressor support &lt;format&gt; on &lt;platform&gt; at &lt;version&gt;?" question for every codec the library documents. Rows carry `Format in`, `Format out`, `Android min-API`, `iOS min-version`, `Codec path`, `Fast-path (Android)`, `Fast-path (iOS)`, and `Notes` columns — fast-path is tracked per platform because Android Media3 Transformer always re-encodes video while iOS `AVAssetExportSession` can bit-stream copy. Authoritatively covers JPEG / PNG / WEBP / HEIC / HEIF / AVIF / GIF / BMP / DNG (image), AAC / MP3 / FLAC / OGG / Opus / AMR-NB / WAV (audio), H.264 / H.265 / VP9 / AV1 (video). Source-of-truth lives in a new `internal object FormatSupportMatrix` (`commonMain`) with a pure-Kotlin renderer; `FormatSupportMatrixConsistencyTest` (`commonTest`) cross-references every row against `ImageFormatGates` constants, the `MIME_AUDIO_AAC` / `MIME_VIDEO_H264` / `MIME_VIDEO_HEVC` constants consumed by `Supportability.evaluateSupport`, and the per-platform fast-path contract (no Android fast-path on video; iOS-unsupported rows cannot claim iOS fast-path); `FormatSupportMatrixBuildVersionPinTest` (`androidHostTest`) pins `ANDROID_MIN_SDK` / `IOS_MIN_VERSION` to `gradle/libs.versions.toml`, and `FormatSupportDocUpToDateTest` (`androidHostTest`) asserts the committed markdown is byte-identical to the renderer output. A new `scripts/regenerate-format-support-doc.sh` wrapper rewrites the file via `-PregenerateFormatSupportDoc=true`; a dedicated `.github/workflows/format-support-check.yml` runs the up-to-date + pin tests without the `docs/` paths-ignore filter so doc-only PRs cannot sneak a stale matrix through. README "Formats" section gains a pointer to the new matrix [CRA-43]

### Changed

* **api (breaking, pre-1.0):** split `CodecSupport` into a `sealed interface` with `Video` and `Audio` `data class` variants. The previous 12-field flat shape had half the fields nulled-by-default per media type (e.g. `audioSampleRates` on video codecs, `maxResolution` on audio codecs) and the type signature was lying about applicability. `DeviceCapabilities.video: List<CodecSupport.Video>` and `DeviceCapabilities.audio: List<CodecSupport.Audio>` now carry the typed split through, three internal helpers in `Supportability.kt` narrow from `CodecSupport` to `CodecSupport.Video` as a bonus. **Migration:** consumers filtering `caps.video[i]` / `caps.audio[i]` keep working unchanged (the filter returns the typed list). If you stored raw `CodecSupport` in your own types, replace with `CodecSupport.Video` / `CodecSupport.Audio` (or keep the sealed interface if you genuinely hold both — smart-cast to read variant-specific fields). Rename `audioSampleRates` → `sampleRates` and `audioMaxChannels` → `maxChannels` at all read sites on audio entries.
* **api (breaking, pre-1.0):** retire the single-value `AudioCodec` enum. Both platform implementations already hard-coded AAC-LC in an M4A container — the enum's `AAC` entry was the only reachable value. Removed: top-level `co.crackn.kompressor.AudioCodec` class; `codec` parameter on `AudioCompressionConfig`; `audioCodec` parameter on `VideoCompressionConfig`; `require(config.codec == AudioCodec.AAC)` guards; `-keep class co.crackn.kompressor.AudioCodec` ProGuard rule. **Migration:** drop `codec = AudioCodec.AAC` from any `AudioCompressionConfig(...)` call-site and `audioCodec = AudioCodec.AAC` from any `VideoCompressionConfig(...)` call-site. Output behaviour is unchanged. A future codec (e.g. Opus on Android / iOS 17+) will introduce its own typed parameter + capability probe rather than re-populate a single-value enum.
* **internal (ios):** extract the shared AVFoundation + POSIX NSError classifier out of `IosAudioErrorMapping.kt` and `IosVideoErrorMapping.kt` into `internal/IosNsErrorClassification.kt`. The two mappers carried byte-for-byte identical `AV_ERR_*` / `POSIX_*` tables under an aspirational \"could diverge\" comment that never materialised — Apple documents these codes once in `AVError.h` with the same semantics across media types. Each mapper shrinks 102→38 / 105→38 lines and now only holds the band→typed-error `when`. Escape hatch if genuine divergence ever appears: add a branch in the mapper's local `when` before deferring to `classifyNSError`. Behaviour unchanged; public `mapToAudioError` / `mapToVideoError` signatures unchanged.
* **docs/api:** expand `docs/api-inventory.md` into the full 1.0 S/E/I freeze list. Every line of `kompressor/api/kompressor.api` is now classified Stable or Experimental with a justification; Internal symbols (absent from the dump by construction) are enumerated separately as a review trail. No symbol was reclassified — the existing ABI dump was audited and every entry stands as intentional public surface. Adds a maintainer sign-off checklist at the bottom of the doc so the 1.0 freeze PR carries explicit two-approver attestation [CRA-17]
* **audio:** zero out iOS surround (5.1/7.1) AAC caps — Device Farm run 24536970778 (iPhone 13 / A15 / iOS 18) confirmed AudioToolbox rejects multichannel AAC output at every tested bitrate (32k–1280k). `iosAacMaxBitrate` / `iosAacMinBitrate` now return 0 for ≥3 channels, and `checkSupportedIosBitrate` surfaces `UnsupportedConfiguration` instead of letting the request reach the hardware encoder probe. Surround AAC remains supported on Android [CRA-82, CRA-78]
* **audio/ios:** encode mono-at-44.1 kHz AAC-LC non-linear cap — the same Device Farm run ([run 24536970778](https://github.com/cracknco/kompressor/actions/runs/24536970778)) found AudioToolbox allocates a larger VBR budget to single-channel streams at 44.1 kHz than the linear per-channel model predicts: up to **256 kbps** accepted (60% above the old 160 kbps cap). `iosAacMaxBitrate` adds a targeted `sampleRate == 44_100` override; other rates retain the linear model. `docs/audio-bitrate-matrix.md` spliced with the full empirical Y/N grid [CRA-78]

### Fixed

* **video (Android):** harden the HDR10 / HEVC Main10 active probe against empty-drain false-negatives (`Hdr10HevcProbe`, CRA-88 follow-up to CRA-20). Replaces the original single-frame-with-EOS protocol — which queued `BUFFER_FLAG_END_OF_STREAM` on the very first (and only) input buffer, triggering an empty output drain on several OMX/Codec2 encoders that need at least one non-EOS frame to bootstrap their lookahead buffer — with a two-frame protocol: queue one priming (non-EOS) P010 frame, drain, and only feed a second EOS-flagged frame when the encoder signals `INFO_TRY_AGAIN_LATER`. Also bumps the probe input dimensions from 1×1 to 16×16 to clear a separate compat class of OEM encoders that reject sub-16px inputs outright. Observable API unchanged — this is a reliability fix hiding behind `probeHdr10HevcSupport()`; devices that were caching a false-negative will re-probe automatically on their next firmware bump via the existing `Build.VERSION.INCREMENTAL` cache key. Docs updated in `docs/guides/probe-and-compatibility.mdx` [CRA-88]
* **ci:** add `CUSTOMER_ARTIFACT` type to Device Farm artifact download in `ios-audio-characterization.yml` and unzip before scanning — XCTAttachments are packaged inside CUSTOMER_ARTIFACT zips, not as FILE type [CRA-82]

### Fixed

* **errors:** iOS `VideoCompressor.compress` no longer leaks an untyped `IllegalArgumentException` from the "no video track found in input file" path — this pre-condition now surfaces as the typed `VideoCompressionError.UnsupportedSourceFormat` that Android has returned all along, so KMP callers get consistent `when`-branchable failures. Same treatment for a file-size race on the happy path of both iOS compressors: the post-pipeline size read now goes through `sizeOrTypedError`, preventing a stray `IllegalStateException("Cannot read file size")` from escaping if the output file disappears between `writer.finishWriting()` and the size probe. `AudioCompressor.compress` KDoc now documents the one remaining `IllegalArgumentException` surface (programmer-error config, e.g. non-AAC codec) [CRA-21]
* **video:** tighten Android HDR10 pre-flight to also require `MediaCodecInfo.CodecCapabilities.FEATURE_HdrEditing` on API 33+ — aligns with Media3's internal `HDR_MODE_KEEP_HDR` gate, so devices that advertise HEVC Main10 but lack the feature now surface `VideoCompressionError.UnsupportedSourceFormat` instead of silently tone-mapping to SDR BT.709 [CRA-6]
* **video:** catch ObjC NSException from AVAssetWriterInput.init via cinterop @try/@catch — prevents crash when HEVC Main10 output settings are rejected by the hardware encoder [CRA-7]
